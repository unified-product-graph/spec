# `properties/`: What Things Carry

Per-entity-type property definitions. For every entity in the catalog, this directory declares the typed fields that entity may carry: start dates, owners, priorities, scored assessments, lifecycle phases, and the rest.

Imports from `shapes/` and `grammar/`.

## Exports

```ts
import {
  // primitives
  type Priority,
  type HealthStatus,
  type Confidence,
  type ISODate,
  type ISODateTime,
  type UPGAssessment,
  // typed property map
  type UPGPropertyMap,
  type UPGNode,
  // runtime schema
  UPG_PROPERTY_SCHEMA,
} from '@unified-product-graph/core'
```

## Files

| File | Role |
|------|------|
| `primitives.ts` | Shared primitive types: `Priority`, `HealthStatus`, `Confidence`, `ISODate`, `ISODateTime`. Re-exports `UPGAssessment` from `../grammar/scales.ts`. |
| `property-map.ts` | `UPGPropertyMap`: master interface mapping every entity name to its property interface. `UPGNode<T>`: typed wrapper combining `UPGBaseNode` with per-type properties. |
| `property-schema.ts` | `UPG_PROPERTY_SCHEMA`: the auto-generated runtime schema. Generated by `scripts/generate-property-registry.ts`, never hand-edited. |
| `domains/` | One file per domain. Each file exports the typed property interfaces for the entities in that domain (e.g. `users.ts` exports `PersonaProperties`, `JobProperties`). |
| `PROPERTIES.md` | Canonical property shapes for cross-cutting keys: when to use `ISODate` vs `ISODateTime`, when `Priority` vs `HealthStatus`, when `Confidence` vs `UPGAssessment`. |
| `status-convention.md` | The `*_status` convention: lifecycle vs runtime state vs event outcome. Read before adding any `*_status` field. |

## Mental model

Properties answers *"when I create a persona, what fields can I fill in?"* The catalog tells you the entity exists. Properties tells you what shape its body takes.

## Start here

1. `PROPERTIES.md`: read the canonical shapes for cross-cutting keys before adding any new field.
2. `status-convention.md`: read before naming any `*_status`, `*_state`, or `*_outcome` field.
3. `primitives.ts`: the small set of primitive types every domain file reaches for.
4. `domains/`: the per-domain property interfaces.

## Layering

The canonical property schema stays narrow. Frameworks (in `frameworks/`) self-declare any additional fields their scoring needs. These framework-scoped fields live inside the framework, not on the entity. See `../ARCHITECTURE.md` (Framework Properties section) for the consumer contract.
