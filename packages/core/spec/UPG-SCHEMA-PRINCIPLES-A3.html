Unified Product Graph Schema Principles v0.2

unifiedproductgraph.org
P1 Readability Over Brevity
Entity type names must be human-readable and self-explanatory. A person encountering the schema for the first time should understand what each type represents without consulting a glossary.
Test: Can a non-specialist read the type name and understand what kind of thing it represents?
P2 Framework-Neutral Naming
Entity types represent universal product concepts, not framework-specific terminology. When a concept exists across multiple frameworks, the canonical name is the most universal descriptor. Framework-specific names are display labels.
Test: Does this name make sense outside any single framework?
P3 Complete Property Schemas
Every entity type must have a typed property interface. Property schemas define the structured knowledge each entity carries. This is UPG's core value proposition: typed, validated, queryable properties.
Test: Can a tool render a meaningful form or summary without guessing what fields exist?
P4 Hierarchy Is Grammar
Valid parent-child relationships are a first-class concept. Every entity type declares its valid parents and children. Self-nesting is explicitly allowed or denied. Vocabulary without grammar is a dictionary, not a specification.
Test: Can a tool validate a .upg file's tree structure using only the spec?
P5 Edges Must Have Verbs
Every edge type carries a human-readable verb describing the relationship. The _has_ pattern is acceptable only for strict containment. All other relationships use semantic verbs.
Test: Can a reader understand the relationship by reading the edge verb alone?
P6 Domains Are Semantic Boundaries
Domains represent distinct areas of product knowledge with internal coherence. Types within a domain relate more to each other than to external types. Cross-domain relationships are expressed through edges.
Test: Can you explain this domain in one sentence? Do all its types belong?
P7 One Concept, One Type
No two entity types represent the same underlying concept. Overlapping types are consolidated into one with a discriminator property. Framework-specific variants are display labels, not separate types.
Test: If you removed one of these types, would users lose the ability to express something?
P8 The Spec Is Canonical
The UPG specification is the single source of truth. Applications derive from it. When they conflict, the spec wins. When the spec is silent, a spec decision is made first, then implemented.
Test: If you regenerated app code from the spec, would any product knowledge be lost?
P9 Naming Follows Domain Language
Within each domain, naming follows established practitioner vocabulary, but only when that vocabulary is universal to the domain, not specific to one framework or tool.
Test: Would a practitioner recognise this term without needing a specific book?
P10 Research Before Definition
No domain is defined by guessing. Properties are researched against existing tools, practitioner needs, AI context utility, and queryability before being added to the spec.
Test: Can you cite a source for every property in this schema?
P11 App Extensions Are Additive
Applications may extend entity types with app-specific properties but must never contradict the spec, rename a property, or promote optional to required. Extensions live alongside spec fields and are preserved during import/export.
Test: Strip all app extensions. Are the remaining properties still a valid UPG node?
P12 JSDoc Is Documentation
Every property has a JSDoc comment explaining what it is and what its values mean. The spec is self-documenting. Import the types, get the docs in your editor. No separate docs page required.
Test: Can a developer understand every property without leaving their IDE?
P13 Shared Vocabulary Axes
When the same conceptual axis applies across entity types, it uses the same values and vocabulary. The axis is shared, not duplicated. Field names may differ by context, but value sets are identical.
Test: If two properties accept the same values, are those values defined once?
P14 Foreign Keys Are Edges
Relationships between entities are expressed through edges, not foreign key properties. The spec uses edges exclusively. App-level foreign keys are an optimisation, not canonical schema.
Test: Does any property reference another entity's ID? That should be an edge.
P15 Two-Level Lifecycle
Lifecycles have two levels: phases (universal, fixed by the spec) and states (specific, extensible by tools within phases). Round-trip fidelity is guaranteed across tools.
Test: Export, import into another tool, re-import. Zero status data loss?
P16 Machine-Written, Human-Readable
The format is produced by agents and tools, consumed by AI and rendering engines. Self-describing over compact. Every value carries enough context to interpret without external definitions.
Test: Can an AI build a useful summary from this node without external schema files?
P17 Qualitative Assessments
Human judgments carry both a numeric value (for computation) and a qualitative label (for display). Scale definitions provide mappings. Computed scores are normalized to 0-1 for cross-tool comparison.
Test: Can a UI show a label without the number? Can a formula use the number without the label?
P18 Bidirectional Edge Verbs
Every edge carries a forward verb (source to target) and a reverse verb (target to source). Data stores one direction. Display is context-dependent. One edge row, two display labels.
Test: Can both source and target render a natural-language description of this relationship?
P19 Referenceable Values Are Entities
When a property value needs to be referenced, grouped by, or shared across multiple entities, it should be an entity type with edges, not an inline property. If two entities need to agree on a value, that value is a relationship.
Test: Could another entity need to point at this value? If yes, it's an entity.
